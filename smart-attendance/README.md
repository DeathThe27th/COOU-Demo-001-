# Omnipresent by COOU — Smart Attendance Using Facial Recognition

Final-year Computer Science project — Chukwuemeka Odumegwu Ojukwu University (COOU), Uli.
Supervisor: Prof. I. J. Mgbeafulike.

A Flask web application that records class attendance by **1:1 face verification**:
the student types their matric number, and the system verifies their live webcam
face against **only that student's** enrolled face embeddings (SFace, 128-d,
cosine similarity). This is O(1) per verification regardless of how many students
are enrolled — adding student #501 requires no retraining of anything.

## Tech stack

| Layer | Technology |
|---|---|
| Backend | Python 3.10+, Flask |
| Face detection | YuNet (`cv2.FaceDetectorYN`, ONNX) |
| Face recognition | SFace (`cv2.FaceRecognizerSF`, ONNX) — 128-d embeddings + cosine similarity |
| Database | SQLite (offline demo) · PostgreSQL/Supabase (hosted) |
| Frontend | HTML/CSS/vanilla JS (Jinja2), browser `getUserMedia` webcam capture |
| Export | pandas + openpyxl (.xlsx) |
| Hosting | Render (container) + Supabase (Postgres) |

## Setup (one-time, needs internet)

```bash
pip install -r requirements.txt
python setup_models.py        # downloads the two ONNX models (~40 MB total)
```

`setup_models.py` fetches from the official OpenCV model zoo:

- `face_detection_yunet_2023mar.onnx` → `models/face_detection_yunet.onnx`
- `face_recognition_sface_2021dec.onnx` → `models/face_recognition_sface.onnx`

**This is the only step that needs internet.** After it completes, the entire
system — enrollment, verification, reporting — runs fully offline.

## Run

```bash
python app.py
# open http://localhost:5000
```

The database is created and seeded automatically on first run.

> **Webcam note:** browsers only allow camera access on `localhost` or HTTPS.
> On the demo laptop use `http://localhost:5000` directly.

### Seeded accounts

| Role | Username | Password |
|---|---|---|
| Admin | `admin` | `admin123` |
| Lecturer | `lecturer1` | `lecturer123` |
| Students | matric no (e.g. `2021/CS/001`) | `student123` |

Five demo students (2021/CS/001 … 005) are pre-seeded **without face data** —
enroll their faces live via *Students → Capture face* (5–8 shots each).

## Demo walkthrough

1. **Admin** logs in → *Students* → capture 5–8 webcam shots per student.
2. **Lecturer** logs in → *Dashboard* → *Open session* for a course → *Open attendance kiosk*.
3. Each **student** at the kiosk: type matric number → *Continue*
   (if already marked present, it says so and the camera never opens) →
   face the camera → *Verify my face* → attendance logged with timestamp in ~2–3 s.
4. Wrong/unenrolled face at a matric number fails; after 3 failed attempts the
   record is flagged `flagged_manual_review` for the lecturer/admin to resolve
   under *Corrections*.
5. Lecturer watches the session table update live, then *Reports → Download .xlsx*
   (rows = students, columns = sessions, cells = Present/Absent, trailing
   Attendance % column).
6. Entire demo runs with wifi off.

## Deployment (Render + Supabase)

The offline demo and the hosted deployment are the **same codebase**, switched by
a single environment variable:

| `DATABASE_URL` | Backend | Used for |
|---|---|---|
| unset | SQLite (`database/attendance.db`) | the offline demo — no network, no config |
| set | PostgreSQL | the hosted deployment |

Hosting needs Postgres because a container filesystem is ephemeral: a local
SQLite file is wiped on every redeploy and every cold start, taking all enrolled
face data with it.

### 1. Supabase

Create a project, then copy **Project Settings → Database → Connection string →
Session pooler**.

Two things that will otherwise cost you an afternoon:

- Use the **Session pooler** string, not *Direct connection*. The direct host
  (`db.<ref>.supabase.co`) is **IPv6-only** and will not resolve from Render.
  The failure looks like a credentials error but isn't.
- **Percent-encode reserved characters in the password**, or the URI parser
  splits the string at the wrong place: `@`→`%40`, `:`→`%3A`, `/`→`%2F`,
  `?`→`%3F`, `#`→`%23`. A password of `pa@@word` must appear as `pa%40%40word`.

### 2. Render

`render.yaml` at the repo root is a Blueprint — *New → Blueprint*, pick this
repo, and Render reads the build command, start command and region from it.

Set one variable by hand in the **Environment** tab:

```
DATABASE_URL = postgresql://postgres.<ref>:<encoded-password>@aws-0-<region>.pooler.supabase.com:5432/postgres
```

`SECRET_KEY` is generated by Render automatically.

The build runs `setup_models.py`, which downloads the two ONNX files at build
time — they are gitignored (38 MB) and never live in the repo.

Notes:

- **Free tier sleeps after 15 minutes idle.** The first request afterwards takes
  ~30 s to wake the container. Fine for a demo, worth knowing before a live one.
- Measured peak RSS with OpenCV, pandas and both models resident is **~206 MB**,
  inside the free tier's 512 MB.
- **The hosted version is not a substitute for the offline demo.** `build.md`
  requires the whole thing to run with wifi off; that is the SQLite path, and it
  keeps working untouched.

### Local development against Postgres

```bash
cp .env.example .env     # fill in DATABASE_URL
python app.py
```

Delete `.env` (or leave `DATABASE_URL` unset) to fall back to SQLite.

## Tuning the similarity threshold

Editable at *Admin → Settings* (stored in the DB, no restart needed).
Start at **0.5** and tune against your real test group:

- Genuine students being rejected (false rejects) → lower it.
  OpenCV's published benchmark threshold for SFace cosine similarity is **0.363**.
- Wrong faces being accepted (false accepts) → raise it.

## Tests

```bash
pip install pytest
pytest tests/ -v
```

Verification logic is tested with synthetic embeddings (no webcam needed);
one test exercises the real ONNX pipeline when the model files are present.

## Known limitations (by design, stated in the project report)

- **No liveness / anti-spoofing detection** — a printed photo may pass. This is a
  stated limitation of a 2D RGB appearance-based system, distinct from
  depth-based systems (Windows Hello / Face ID).
- Requires 5–8 clear, well-lit enrollment shots per student.
- Cosine threshold needs empirical tuning per deployment environment/camera.

### How one codebase targets two databases

All of it is contained in `modules/db.py`; nothing above that layer knows which
backend is live.

- **`Row`** is a case-insensitive mapping. SQLite echoes column names back with
  the casing used in the query (`MatricNo`); Postgres folds unquoted identifiers
  to lowercase (`matricno`). Templates keep writing `{{ s.MatricNo }}`.
- **`_to_pg`** rewrites the SQLite-only constructs actually used here: `?`
  placeholders → `%s`, `INSERT OR IGNORE` → `ON CONFLICT DO NOTHING`, and the
  reserved word `User` → `"User"`.
- **`execute_db`** emulates SQLite's `lastrowid` with a `RETURNING` clause.
- Session/attendance timestamps are **TEXT in both schemas**. The templates slice
  them as strings (`{{ s.StartTime[11:16] }}`) and compare them as strings; real
  `TIMESTAMP` columns would return `datetime` objects and break both. ISO-8601
  sorts lexicographically, and all date arithmetic already happens in Python.
